# SEO Audit Log — Ndauwo Safari Co.

## 2026-05-14 — Initial Audit (Pre-Migration)

### Status
- Frontend (Vue 3 SPA) and backend (Laravel 12 API) are separate projects
- Vue SPA renders entirely client-side — no server-rendered content
- Search engines see only bare HTML shell with no meta tags
- No structured data, canonical URLs, or Open Graph tags

### Plan
Approved plan to unify frontend into Laravel as a single project:
1. Move all Vue source into `backend/resources/js/`
2. Create Blade entry point with server-injected SEO meta
3. Build `SeoService` for route-based meta generation
4. Add JSON-LD, canonical, hreflang support
5. Update all API calls to relative `/api/` paths
6. Keep all Vue components, styles, and animations intact

See `/plans/laravel-unification-plan.md` for full details.

### 2026-05-14 — Phase 4: Database Data Import (MySQL → SQLite)

#### Context
- The original project used MySQL with `database.sql` dump containing all production data
- This machine's MySQL 9.6.0 is corrupted (abort trap 6), preventing MySQL startup
- Switched to SQLite (`DB_CONNECTION=sqlite`) as the working database
- Migrations ran successfully creating all 28 tables in SQLite
- The `database.sql` MySQL dump contains INSERT statements with different column counts/orders than the SQLite schema (due to migration history differences)

#### Approach (Final — PHP Artisan Command)
- Created `backend/app/Console/Commands/ImportMysqlDump.php` — a Laravel artisan command (`db:import-mysql-dump`) that:
  1. Parses MySQL CREATE TABLE statements to get original column names/order
  2. Reads SQLite PRAGMA table_info to get current column names/order
  3. Extracts VALUES blocks using character-by-character scanning (paren-depth counting + string-awareness) to correctly parse multi-row INSERTs containing semicolons inside JSON text values
  4. Parses each row with full MySQL escape sequence handling (`\'`, `\"`, `\\`, `\n`, `\r`, `\t`, `''` doubled quotes)
  5. Maps MySQL columns → SQLite columns via array_search, inserting only columns that exist in SQLite
  6. Skips tables already populated (idempotent)

#### Issues Encountered & Resolved
| Issue | Root Cause | Fix |
|-------|-----------|-----|
| `sqlite3 < import.sql` loaded data positionally | MySQL dump column order ≠ SQLite schema column order | Built explicit column mapping via `getMysqlColumns()` + PRAGMA table_info |
| Python csv module lost rows with complex JSON | Multi-row INSERT parser broke on nested `{}`, `[]`, and `;` inside JSON strings | Replaced with PHP character-by-character `extractValuesBlock()` that scans for terminating `;` outside any string |
| Regex `(.+?);` truncated VALUES block | Semicolons inside JSON text values (escaped as `;`) caused premature match | `extractValuesBlock()` ignores semicolons when inside quoted strings |
| Initial paren-depth counter returned only 1 row | Parent counter hit depth 0 at first row's closing `)` instead of finding the terminating `;` after all rows | Rewrote to scan for `;` at depth 0, ignoring all `)` that close individual rows |
| `App\\Models\\User` escape in dump | MySQL dump uses `\\` which SQLite treats literally | Fixed with `fix_import.py` (transient), then superseded by PHP command |

### 2026-05-14 — Phase 5: Asilia-Inspired Typography System

#### Source Analysis
Analyzed the font system at https://www.asiliaafrica.com/ by fetching their theme CSS (`/wp-content/themes/asilia/css/styles.css`). Their typography architecture:

| Asilia Font | Role | Weight/Style | Free Alternative |
|---|---|---|---|
| **PP Fragment** | Heading serif (h1–h6) | Regular (400), Bold (700) | Playfair Display |
| **Verlag** | Body geometric sans | Book (400), Bold (700), Black (900) | Jost |
| **Sunday April** | Script accent (em in headings) | Regular (400) | Dancing Script |
| **Helvetica Neue** | UI fallback | Normal | System font stack |

Key design decisions from Asilia:
- Headings: `font-family: 'PP Fragment'; font-weight:normal; padding:0 0 25px` — no bold by default
- Script: `h1 em, h2 em, h3 em { font-family:'Sunday April'; font-size:1.4em }` — only for `<em>` inside headings
- Body: `font-weight:400` on `p` (not 300), `line-height:150%`
- Blockquote: PP Fragment at `1.5em` with Unicode curly quotes (`\201C`/`\201D`)
- Heading scale: h1 `3.5em`, h2 `2em`, h3 `1.5em`, h4 `1.3em`, h5 `1.2em`, h6 `1.1em`
- Editor utilities: `.h1-large` (64px), `.h1-medium` (51px), `.h1-small` (40px), etc.
- Selection: `#E7E2D4` (warm beige highlight)
- Background: `#F5F2ED` (warm cream)

### Phase 10: Browser Console Error Fixes (2026-05-15)

#### Issues Reported
1. **MIME type errors**: Images (`/image-XX.jpg`) loaded as `<script type="module">` in production HTML, causing "Expected a JavaScript-or-Wasm module script but the server responded with a MIME type of 'image/jpeg'" errors.
2. **Font preload warnings**: Three `<link rel="preload">` tags for Google Fonts with `as="style"` not consumed within a few seconds, causing console warnings.

#### Root Cause Analysis

**Issue 1 — Image MIME errors:**
The original `externalize-absolute-assets` Vite plugin used `resolveId` with `external: true`. This told Rollup to emit `<script type="module" src="/image-12.jpg">` tags in production HTML. The browser loaded these JPEGs as JavaScript modules and rejected the `image/jpeg` MIME type.

**Issue 2 — Font preload warnings:**
Three separate `<link rel="preload" as="style">` tags (lines 44-50 in `app.blade.php`) each preloaded a single Google Font family. The actual `<link rel="stylesheet">` at line 52-55 combined all four fonts into a single URL. The browser flagged the preloads as unused because their URLs didn't match the stylesheet URL.

#### Changes Applied

**`backend/vite.config.js`:**
- Removed the `externalize-absolute-assets` plugin entirely
- Added `resolve-public-assets` virtual module plugin that:
  - Intercepts `resolveId` for `/image-*`, `/ndauwologo.png`, `/favicon.ico`, `/vite.svg`
  - Returns virtual module IDs (`\0virtual:public-asset:...`)
  - In `load`, returns `export default "/image-XX.jpg"` — a string literal default export
- This causes Rollup to inline the path as a JavaScript string constant instead of generating an `import` statement

**Why `build.rollupOptions.external` didn't work:** Marking images as Rollup external still generates `import X from "/image-12.jpg"` statements in the bundle. The browser fetches these as ESM imports, causing the same MIME error. Only virtual modules that export string literals eliminate the import statements entirely.

**`backend/resources/views/app.blade.php`:**
- Removed lines 44-50: three `<link rel="preload" as="style">` tags for Playfair Display, Jost, and Dancing Script
- The combined `<link rel="stylesheet">` already loads all four families in one request

#### Files Modified
- `backend/vite.config.js` — replaced `externalize-absolute-assets` plugin with `resolve-public-assets` virtual module plugin
- `backend/resources/views/app.blade.php` — removed 7 lines of redundant font preload tags

#### Build Verification
- `npm run build` succeeded: 1,883 modules transformed, 18 output chunks
- `curl http://localhost:8000` confirmed production HTML output:
  - Only one `<script type="module" src=".../main-Ch-HgV4V.js">` tag (the actual app bundle)
  - Zero `<script type="module" src="/image-XX.jpg">` tags
  - Zero font preload `<link>` tags
  - Single combined Google Fonts `<link rel="stylesheet">`
  - All SEO meta tags intact
- `grep "import.*from.*image-"` on the built JS bundle returned **zero matches**

#### Expected Impact
- **Zero MIME type errors** in browser console — no image import statements remain in the bundle
- **Zero font preload warnings** in browser console — only the combined stylesheet link remains

### Phase 11: Fix `/build/` 404 Error + Custom PHP Router (2026-05-15)

#### Issues Reported
1. `GET http://localhost:8000/build/ 404 (Not Found)` — repeating every few seconds in terminal logs
2. `content-script.js:104 Failed to get subsystem status for purpose {rejected: true, message: 'UNSUPPORTED_OS'}` — separate, unrelated browser extension error

#### Root Cause Analysis

**Issue 1 — `/build/` 404:**

PHP's built-in development server (`php -S`) intercepts `GET /build/` before Laravel because:
1. `public/build/` is a real directory on disk (created by `vite build`)
2. The directory has no index file (`index.php` or `index.html`)
3. PHP's built-in server returns its own 404 HTML (`<h1>Not Found</h1>`) without ever reaching Laravel's catch-all route in `routes/web.php`

The requester is **Chrome DevTools** (Network/Sources tab), which probes the `/build/` directory seen in `<link>` and `<script>` tags. This is confirmed by the consistent pairing: `/build/` → `.well-known/appspecific/com.chrome.devtools.json` → `/favicon.ico`.

Individual assets (`/build/assets/main-Ch-HgV4V.js`, `/build/assets/app-DuzDPs8w.css`) returned HTTP 200 — only the bare directory path `/build/` 404'd.

**Issue 2 — `content-script.js` `UNSUPPORTED_OS`:**

Completely unrelated — this is a browser extension (not part of the Ndauwo codebase) failing to get OS subsystem status. No action needed.

#### Changes Applied

**`backend/server.php`** — Custom PHP built-in server router (NEW file):

Created a PHP router script that ensures ALL requests go through Laravel's front controller, not PHP's default handler:

```php
$uri = urldecode(parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH));
$publicPath = __DIR__ . '/public' . $uri;

// Only let PHP serve actual FILES (CSS, JS, images) directly.
// Directories (like /build/) MUST go through Laravel.
if ($uri !== '/' && is_file($publicPath)) {
    return false;  // PHP serves natively
}

// Everything else → Laravel's index.php
require_once __DIR__ . '/public/index.php';
```

**Key insight**: The fix uses `is_file()` not `file_exists()`. `file_exists()` returns `true` for both files AND directories, which is the bug we're fixing. `is_file()` returns `true` only for actual files, routing all directory requests (like `/build/`) through Laravel.

#### How to Run the Server Now

```bash
cd backend && php -S localhost:8000 -t public server.php
```

Note: Laravel 12's `php artisan serve` does not support the `--router` option (removed in Laravel 11), so we use PHP's built-in server directly with the custom router script.

| CSS asset | `/build/assets/app-DuzDPs8w.css` | 200 | 200 |

All existing functionality preserved — no regressions.
